Kontek Design System · Mobile

Bottom Tab Bar Navigation

The primary mobile navigation pattern. Four core areas plus a raised center action, permanently in thumb reach — the replacement for side navigation (sidonavigering) on phones.

Component spec iPhone + Android v1.0
2 Home
Pay
3 Notifications
Profile

Resting state — Home active, count badges on Home and Notifications, raised "+" center action.

1Why this replaces side navigation

Side navigation hides the app's main areas behind a menu the user has to open. On a phone that costs a tap for every move and pushes the most-used destinations out of sight.

This pattern puts the four core areas plus the primary "register" action permanently in reach of the thumb, so the whole app is one tap away and the thing people open the app to do — register absence or an expense — is always the visible center action.

Use a bottom tab bar for the top-level areas of the app. Keep side navigation only for rare, secondary destinations, if at all.

2Anatomy

A fixed bar pinned to the bottom of the screen, above the safe-area inset. Five equal slots:

SlotTypePurpose
1TabHome — overview / dashboard
2TabPay — pay slips
3Center actionRegister — raised "+" that opens the action sheet
4TabNotifications
5TabProfile
  • Max 5 slots. Four tabs + one center action is the ceiling. Never add a sixth.
  • The center action is not a tab — it does not represent a destination, it triggers an action sheet. It never shows a selected state.

6Count badge

For tabs that signal pending items — Home (number of "to do" items), Notifications (unread count).

Shapeperfect circle, 18×18px
Background#61BC8F (--green)
Number color#FFFFFF
Font10px / 700, centered
Positiontop-right of the icon, not the label
Visibilityshown only when count > 0
Overflowcap display at 9+

The badge is a count, not a dot. Use a numbered badge wherever a quantity is meaningful; reserve a plain dot only for "something changed, no count."

7Behavior rules

  • Persistent. Visible on all top-level screens. Hidden inside full-screen flows (registration steps, sheets) and deep detail screens with their own back-nav app bar.
  • One active tab. Switching tabs resets that area to its root (clears its navigation stack).
  • Center action never selects. No active/destination state — only resting vs. open.
  • No horizontal scrolling. If you need more than 4 destinations, the IA is wrong — consolidate, don't scroll the bar.
  • Labels always shown. Never icon-only; the 11px label is part of the pattern.

8Accessibility

  • Every tab: aria-label = its name, aria-current="page" when active.
  • Center action: aria-label="Register", aria-expanded reflects sheet state.
  • Tap targets fill the full slot height; minimum 48×48px effective.
  • Color is never the only signal of the current area — icon + label pairing and screen content reinforce it. Active label #203B3C on white is AAA for text.

10Do / Don't

Do
  • Keep it to 4 tabs + 1 center action.
  • Use color-only selection for tabs.
  • Put the primary create action in the raised center slot.
  • Show numbered badges for pending counts.
Don't
  • Don't reintroduce a hamburger / side drawer for top-level areas.
  • Don't give the "+" a selected/destination state.
  • Don't add pills, fills, or underlines to the active tab.
  • Don't hide labels, let them wrap, or exceed 5 slots.
